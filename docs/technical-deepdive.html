01 — The Big Picture

End-to-End Request Flow

When a user asks "Show my spending trends", this is exactly what happens — from the browser to the LLM and back. Every box is a real function in the codebase.

graph TD USER["👤 User types query"] --> FE["🖥️ Frontend — static/index.html\nPOST /api/query/stream"] FE --> SRV["⚡ server.py\nFastAPI + Rate Limiter + Security Headers"] SRV --> S1["🛡️ STAGE 1 — Input Guardrails\nguardrails.py\n5 sequential checks"] S1 --> S2["📋 STAGE 2 — Context Assembly\ncache.py + pipeline.py\nProfile + history + system prompt"] S2 --> S3["🤖 STAGE 3 — LLM Reasoning\nllm_client.py + pipeline.py"] S3 --> TOOLS{"Response type?"} TOOLS -->|"Tool calls"| VIZ["📊 visualizations.py\n12 Plotly charts → PNG"] TOOLS -->|"Text"| TEXT["💬 Text response"] TOOLS -->|"LLM failed"| FALL["🔄 _fallback_response\n31 intent handlers"] VIZ --> S4["🔒 STAGE 4 — Output Guardrails\nXSS + hallucination + toxicity"] TEXT --> S4 FALL --> S4 S4 --> SSE["📡 SSE Stream → Frontend\nchunk | charts | done"]

Every query passes through all 4 stages. No shortcuts, no bypasses.

🎤 Interview Talking Points

  • "The pipeline is deterministic." — Every request hits all 4 stages in order. Input guardrails always run before the LLM sees anything.
  • "Three response paths." — LLM can return tool calls (charts), plain text, or fail entirely. Each path is handled gracefully.
  • "Zero-downtime fallback." — If all 8 LLM models fail, 31 keyword-based intent handlers generate data-driven responses from the raw DataFrame. The user never sees an error.

  • "SSE (Server-Sent Events) instead of WebSocket — simpler, one-directional, works through proxies."
  • "Charts are encoded as base64 data URIs inline in SSE events — no separate image fetch needed."

03 — Stage 1

Input Guardrails

5 sequential checks run on every prompt before the LLM sees anything. If any check fails, the request is rejected immediately with a safe error message.

graph TD IN["Incoming Prompt"] --> N["1️⃣ Unicode Normalization\nNFKC + strip zero-width chars\n─────────────────\nPrevents: Homoglyph bypass\nExample: ɪgnore → ignore"] N --> INJ["2️⃣ Prompt Injection Check\n18 regex patterns\n─────────────────\nCatches: 'ignore instructions'\n'you are now DAN'\n'system prompt override'"] INJ -->|"❌ Blocked"| R1["Rejection: Injection detected"] INJ -->|"✅ Pass"| LEAK["3️⃣ Cross-User Leakage\nCheck for other user IDs\n─────────────────\nCatches: 'show usr_002 data'\n'another user's spending'"] LEAK -->|"❌ Blocked"| R2["Rejection: Access denied"] LEAK -->|"✅ Pass"| SCOPE["4️⃣ Scope Validation\n100+ financial keywords\n─────────────────\nAllows: spending, budget, groceries\nBlocks: 'write me a poem'"] SCOPE -->|"❌ Blocked"| R3["Rejection: Off-topic"] SCOPE -->|"✅ Pass"| LEN["5️⃣ Length Check\nMax 500 characters\n─────────────────\nTruncates oversized prompts\nPrevents context overflow"] LEN --> OUT["✅ Clean prompt → Stage 2"]

🎤 How to explain Stage 1

  • "Defense in depth." — Unicode normalization runs first because injection patterns won't match if the attacker uses look-alike characters (homoglyphs).
  • "The scope check is smart." — It doesn't just look for financial keywords. Conversational patterns like "what can you do?" always pass through. Helper words + financial keywords also pass ("how much did I spend").
  • "Cross-user leakage is critical for multi-tenant." — With 30 users sharing one pipeline, we check that no prompt references another user's ID or uses phrases like "another user".
04 — Stage 2

Context Assembly

Before calling the LLM, we assemble everything it needs to give a personalized, accurate response — the user's profile, conversation history, and tool schemas.

graph TD START["Clean Prompt"] --> CACHE{"Profile cached?"} CACHE -->|"HIT (within 30min)"| PROF["📋 Cached Profile\nuser_name, top 5 categories\navg monthly spend/income"] CACHE -->|"MISS / expired"| BUILD["🔨 build_profile()\nCompute from DataFrame → Cache"] BUILD --> PROF PROF --> SYS["📝 _build_system_prompt()\n─────────────────────\n• Vola AI persona\n• User's financial profile\n• 12 tool schemas (charts)\n• Response formatting rules\n• Proactive engagement rules"] SYS --> CONV["💬 Conversation History\nLast 6 messages\n(sliding window)"] CONV --> TOKEN["✂️ enforce_token_budget()\nTrim to 4000 tokens\nKeep most recent context"] TOKEN --> MSG["📨 Final Messages Array\n[system_prompt, ...history, user_msg]"] MSG --> NEXT["→ Stage 3: LLM"]

🎤 How to explain Stage 2

  • "The system prompt is ~8K characters." — It includes the user's actual financial data (top categories, avg spend), so the LLM can give personalized answers without querying the DataFrame every time.
  • "Token budget prevents context overflow." — If conversation history + system prompt exceeds 4000 tokens, older messages are trimmed first. The system prompt and latest user message are always preserved.
  • "The cache has TTL + LRU." — Profiles expire after 30 minutes (data freshness). If we hit 100 cached users, the least-recently-used one gets evicted (memory management).
05 — Stage 3

LLM Reasoning & Tool Dispatch

The LLM receives the context, reasons about the query, and either responds with text or invokes visualization tools. If the LLM is unavailable, 31 fallback handlers take over.

graph TD MSG["Messages from Stage 2"] --> INT{"🎯 Intent Intercept?"} INT -->|"'full report'"| RPT["_comprehensive_report()\n7 sections + 8 charts"] INT -->|"'plan a budget'"| BUD["Budget handler\n50/30/20 rule table"] INT -->|"'subscriptions'"| SUB["Subscription handler\nRecurring charge detection"] INT -->|"'better graph'"| REDO["Viz state from cache\n→ Regenerate chart"] INT -->|"Normal query"| LLM["🤖 LLMClient.chat_stream()"] LLM --> RESP{"Response type?"} RESP -->|"tool_calls"| EXEC["_execute_tool_calls()\nMaps name → VIZ_FUNCTIONS\n→ Plotly → PNG"] RESP -->|"Text content"| STRM["Stream text tokens"] RESP -->|"Error / empty"| FALL["_fallback_response()\n31 intent handlers\nData-driven answers"] EXEC --> EXP["_explain_chart()\nData-driven explanation"] EXP --> S4["→ Stage 4"] STRM --> S4 FALL --> S4 RPT --> S4 BUD --> S4 SUB --> S4

🎤 How to explain Stage 3

  • "Intent intercepts bypass the LLM for known patterns." — Budget, subscription, and report requests have specialized handlers that produce better results than a generic LLM call.
  • "Tool calling is how charts happen." — The LLM sees 12 function schemas (TOOL_SCHEMAS). When it returns a tool_call like plot_monthly_spending_trend({months: 6}), we execute the matching function from VIZ_FUNCTIONS.
  • "The fallback system is the safety net." — 31 keyword-based handlers match things like 'budget', 'merchant', 'savings', 'trend', 'heatmap'. Each generates a data-driven response from the DataFrame + relevant charts."

06 — Stage 4

Output Guardrails & Response Delivery

Before the response reaches the user, 4 output checks run. Then follow-up questions are appended, caches are updated, and the response is streamed via SSE.

graph TD IN["Raw LLM Output"] --> XSS["1️⃣ XSS Sanitization\n14 regex patterns\nStrip: script, iframe, onerror, javascript:"] XSS --> HALL["2️⃣ Hallucination Grounding\nExtract all $ amounts from response\nValidate against actual data\n10% tolerance threshold"] HALL --> CONF["3️⃣ Confidence Check\nDetect: 'I'm not sure'\n→ Flag for review"] CONF --> TOX["4️⃣ Toxicity Filter\nKeyword-based scan\nBlock offensive content"] TOX --> FUP["📎 Append Follow-Up\nContextual numbered options\n'Would you like to...\n1. See breakdown\n2. Compare months'"] FUP --> UPD["💾 Update Caches\nProfile + Conversation\nViz state + Query history"] UPD --> LOG["📋 Audit Log\nHashed ID, prompt, latency"] LOG --> SSE["📡 SSE Stream to Client\nchunk → charts → done"]

🎤 How to explain Stage 4

  • "Hallucination grounding is the key innovation." — We extract every dollar amount from the LLM's response, check it against the actual DataFrame summary, and flag anything that's off by more than 10%. This prevents the LLM from inventing numbers.
  • "XSS sanitization on LLM output." — LLMs can be tricked into outputting malicious HTML. We strip 14 categories of dangerous content (script tags, event handlers, iframes, javascript: URIs) from every response.
  • "Every response drives the conversation forward." — _generate_followup_question() analyzes what the user just asked and appends 2-3 contextual numbered options. This keeps users engaged.
07 — LLM Client

BYOK Provider Routing & Fallback Chain

The LLM client supports two providers (Sarvam AI and OpenRouter) with automatic detection, retry logic, exponential backoff, and a chain of 8 models.

graph TD INIT["LLMClient.__init__()"] --> DET{"Auto-detect provider"} DET -->|"SARVAM_API_KEY found"| SAR["Sarvam AI\napi.sarvam.ai/v1\nsarvam-105b → sarvam-30b"] DET -->|"Only OPENROUTER key"| OR["OpenRouter\nopenrouter.ai/api/v1\n6 free models"] DET -->|"No keys"| ERR["❌ Error"] SAR --> CALL["chat_stream(messages, tools)"] OR --> CALL CALL --> TRY["Try Model #1"] TRY --> RES{"Status?"} RES -->|"200 ✅"| OK["Stream tokens\nCircuitBreaker.record_success()"] RES -->|"429 Rate Limited"| SKIP["Skip → next model"] RES -->|"503 Unavailable"| BACK["Backoff 2^n sec → retry"] BACK --> TRY SKIP --> MORE{"More models?"} MORE -->|"Yes"| TRY MORE -->|"No"| FAIL["All failed\n→ _fallback_response()"]

🎤 How to explain the LLM client

  • "BYOK = Bring Your Own Key." — Users can use their own Sarvam AI key (paid, higher quality) or the default OpenRouter key (free tier). The client auto-detects which is available.
  • "Model chain, not single model." — If sarvam-105b fails, it tries sarvam-30b. On OpenRouter, 6 different models from different providers (Nvidia, Google, MiniMax, etc.) ensure at least one is up.
  • "Smart retry." — 429 (rate limited) = skip to next model immediately (waiting won't help). 503 (server down) = exponential backoff and retry (might recover). Other errors = skip to next model.
  • "Circuit breaker prevents cascade." — After 5 consecutive failures, the circuit opens for 60 seconds. During this window, we go straight to fallback handlers instead of wasting time on failing API calls.
08 — Charts

12 Chart Types, One Rendering Pipeline

All 12 charts follow the same pattern: filter user data → build Plotly figure → apply dark theme → export as high-DPI PNG via Kaleido + Chromium.

graph TD A["LLM returns tool_call:\nplot_monthly_spending_trend\n{months: 6}"] --> B["_execute_tool_calls()\nLookup: VIZ_FUNCTIONS dict"] B --> C["plot_monthly_spending_trend(df, user_id, months=6)"] C --> D["_user_df() → filter to user\n_filter_months() → last 6 months"] D --> E["Build Plotly Figure\ngo.Scatter traces\n+ rolling average"] E --> F["_base_layout() dark theme\n#0D0F11 bg, #6C5CE7 accent\nInter font family"] F --> G["_save() → fig.write_image()\n1100×650 px, 2× scale\nKaleido + Chromium engine"] G --> H["Returns: output/user_trend_20260422.png"] H --> I["server.py reads PNG\nEncodes to base64 data URI\nSends as SSE 'charts' event"]

🎤 How to explain the chart system

  • "Server-side rendering, not client-side." — Charts are PNG images generated on the server. The browser just displays an <img> tag. No JavaScript charting library needed on the client.
  • "LLM decides which chart to show." — The 12 tool schemas are in the system prompt. The LLM reasons about the query and returns a tool_call with the right function name and parameters.
  • "Kaleido + Chromium." — Kaleido 1.2.0 uses a headless Chromium browser to render Plotly charts as PNG. In Docker, we install the full Chromium binary and set CHROME_PATH.
  • "Every chart gets an explanation." — _explain_chart() analyzes the chart type and user data to generate a narrative like 'Your spending peaked in March at $2,400'."
09 — Proactive Conversations

How short replies become full queries

Every response ends with numbered options. When the user replies with just "yes", "2", or "groceries", the follow-up resolver converts it into a complete query.

graph TD U["User says: 'yes' or '2' or 'groceries'"] --> CHK{"≤ 5 words?"} CHK -->|"No — full sentence"| PASS["Pass through unchanged"] CHK -->|"Yes"| HIST["Get last assistant message\nfrom conversation cache"] HIST --> TYPE{"Reply type?"} TYPE -->|"'yes' / 'sure'"| YES["LLM rewrites:\n'Show the breakdown\nthe AI just offered'"] TYPE -->|"'1' or '2'"| NUM["Extract option text:\n'2' → 'Show all categories'\n→ query: 'show all categories'"] TYPE -->|"'groceries'"| TOPIC["Rewrite:\n'show groceries transactions'"] TYPE -->|"'no' / 'nah'"| NO["'What would you like\nto explore instead?'"] YES --> FULL["Rewritten query\n→ Pipeline Stage 1"] NUM --> FULL TOPIC --> FULL

🎤 How to explain the follow-up system

  • "Two-part system." — Output side: _generate_followup_question() appends contextual options. Input side: _resolve_followup() converts short replies back into full queries.
  • "Topic words are the clever part." — If the user just types "groceries" (≤5 words, not a number, not yes/no), we rewrite it as "show groceries transactions" which then routes correctly through the category handler.
  • "It creates a natural conversation loop." — Ask → Answer with follow-up → User clicks/types option → Resolve → Ask again. The AI drives the dialogue forward.
10 — Deployment

Docker → Render.com

Multi-stage Docker build. Push to GitHub → Render auto-deploys. Chromium binary included for server-side chart rendering.

graph LR subgraph LOCAL["Local Dev"] L1["python server.py\nlocalhost:8000"] end subgraph DOCKER["Docker Build"] D1["Stage 1: python:3.12-slim\npip install → /install"] D2["Stage 2: python:3.12-slim\n+ Chromium + fonts\n+ app code"] D3["Non-root appuser\nuvicorn --workers 2"] D1 --> D2 --> D3 end subgraph RENDER["Render.com"] R1["GitHub push → auto-deploy"] R2["Docker build"] R3["Health: GET /api/users"] R4["HTTPS live"] R1 --> R2 --> R3 --> R4 end LOCAL -.->|"git push"| RENDER DOCKER -.->|"Used by"| RENDER

🎤 How to explain deployment

  • "Multi-stage Docker." — Stage 1 installs pip packages into a prefix directory. Stage 2 is the clean production image — copies packages + installs Chromium. Smaller final image.
  • "Non-root user." — appuser runs the server. Even if there's an exploit, the attacker can't modify system files.
  • "Chromium in Docker." — Kaleido 1.2.0 needs a real browser binary to render charts. We install the chromium package and set CHROME_PATH env var.
  • "Health check." — Render pings GET /api/users every 30 seconds. If it fails 3 times, the service restarts.

11 — Design Decisions

Questions they might ask

"Why not use a vector database / embeddings for RAG?"

The data is tabular (CSV), not documents. Semantic search over embeddings doesn't help when you need exact aggregations like "total groceries spending in March". Instead, we use the LLM for intent understanding and pandas for computation — a tabular RAG approach where the "retrieval" is SQL-like DataFrame operations.

"Why SSE instead of WebSocket?"

The communication is one-directional: server → client streaming. SSE is simpler (works over HTTP/1.1, auto-reconnects, works through all proxies). WebSocket adds complexity for bidirectional communication we don't need.

"Why server-side chart rendering instead of client-side?"

Three reasons: (1) No JavaScript charting library needed on the client — the frontend is a single HTML file. (2) Charts look identical everywhere — no browser rendering differences. (3) Charts can be cached and reused.

"Why 31 fallback handlers instead of just showing an error?"

Zero-downtime for the user. LLM APIs have ~5% failure rates. With 31 keyword-based handlers + pandas, we can answer every common financial question from raw data. The user never sees "Service unavailable".

"Why in-memory cache instead of Redis?"

For a single-instance deployment on a free tier, in-memory is simpler and faster. The TTL + LRU eviction handles memory management. If we scaled horizontally, we'd add Redis — the cache interface (get/set) wouldn't change.

"How do you prevent hallucination?"

Two layers: (1) The system prompt includes the user's actual financial data, so the LLM has grounded context. (2) Output guardrails extract every dollar amount from the response and validate it against the DataFrame summary with a 10% tolerance. Ungrounded numbers are flagged.

"What's the latency like?"

Typical: 2-4 seconds for text-only responses (LLM streaming starts in ~500ms). Chart responses add ~1-2 seconds for Plotly rendering. The SSE streaming means the user sees text appearing immediately — perceived latency is sub-second.
